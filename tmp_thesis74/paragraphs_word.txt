38	摘  要
39	随着校园线上交流场景由“信息发布”逐步转向“内容互动、趋势洞察与治理协同”，传统论坛以时间倒序为核心的组织方式，已难以同时满足高校社区在内容分发效率、风险控制与数据辅助决策方面的综合需求。围绕这一现实问题，本文结合本人独立实现的 Campus Pulse 项目，设计并实现了一套面向校园内容交流与技术协作场景的智能社区平台。
40	系统采用 Spring Boot 3.5.8、Spring Security、MyBatis-Plus、MySQL、Redis 与 Vue 3 + TypeScript 的前后端分离架构，构建了注册登录、帖子发布、评论回复、点赞收藏、私信通知、板块管理、版主审核、趋势统计与后台运维等完整业务链路。后端通过 JWT 会话令牌、设备标识校验、请求签名与二步验证增强安全性；通过 Redis 版本化缓存、热度衰减任务和 WebSocket 推送提升交互实时性；通过 AI 标签/摘要提取、情感打分和混合推荐机制增强内容理解与分发能力。
41	关键词：校园内容社区；趋势分析；权限控制；混合推荐；实时通知
49	Abstract
50	With campus online communities shifting from simple information boards to interactive content spaces, chronological forum models are no longer sufficient for efficient distribution, risk control, and operational insight. Based on the Campus Pulse project implemented in this study, this paper designs and develops an intelligent community platform for campus-oriented content communication and collaborative governance.
51	The system adopts a front-end/back-end separated architecture built with Spring Boot 3.5.8, Spring Security, MyBatis-Plus, MySQL, Redis, Vue 3 and TypeScript. It provides user registration and login, post publishing, threaded comments, likes, favorites, private messages, notifications, moderator review, trend statistics and administrative management. Security is strengthened through JWT session tokens, device verification, request signatures and TOTP-based two-factor authentication. Real-time responsiveness is improved through Redis versioned caching, scheduled hotness decay and WebSocket push, while AI-assisted tag extraction, summary generation, sentiment scoring and hybrid recommendation enhance content understanding and distribution.
52	Key words: campus community; trend analysis; access control; hybrid recommendation; real-time notification

86	1.1 研究背景与意义
87	近几年，高校信息服务明显走向社区化。课程交流、活动组织、资源共享和经验讨论，都要求在统一平台完成浏览、互动与反馈。与传统论坛相比，校园社区更新更快、话题更碎、反馈更即时，若仍只依赖时间排序和人工巡查，就难以及时发现高价值内容、异常舆情和兴趣变化[1][2][8][9]。
88	本课题的意义体现在应用、技术和实践三个层面。应用上，Campus Pulse 将内容发布与运营决策放入同一系统，让普通用户、版主和管理员围绕统一数据协同。技术上，项目把前后端分离、缓存、实时推送、推荐与安全治理整合到同一工程，验证了中小规模校园社区的实现路径[3][4][8][9]。实践上，系统通过热度衰减、板块审核、举报流转和通知反馈，为高校数字社区治理提供了工程样例。
89	1.2 研究内容
90	本文围绕 Campus Pulse 的真实实现展开，研究内容直接对应代码、数据库与前端页面。核心任务是完成校园社区平台的整体架构设计与业务实现，包括用户认证、内容发布、评论互动、私信通知、板块治理、后台管理和趋势展示等模块，形成可运行的前后端分离系统。
91	其次，针对校园社区高频访问与安全风险并存的特点，系统采用 Spring Security + JWT + Redis 认证链路，并加入设备绑定、请求签名、防重放和 TOTP 二步验证；性能侧则通过版本化缓存、分页查询、定时任务和异步处理保证主要接口响应效率[4][10][14][15]。
92	再次，在内容智能化方面，系统接入外部 AI 服务完成标签提取与摘要生成，在本地实现敏感词过滤、情感分析、热度计算和混合推荐，并用热榜、关键词聚合和趋势预测为管理员提供辅助决策。本文希望说明，校园社区平台不仅能完成功能搭建，还能在真实业务链路中稳定运行并具备扩展空间[3][5][6][7][11][12]。

104	6.Redis缓存
105	Redis缓存是一种高性能的内存键值存储数据库，常用于解决关系型数据库的性能瓶颈。它以极低的延迟（通常亚毫秒级）提供数据读写，支持字符串、哈希、列表、集合、有序集合等多种数据结构。在实际应用中，Redis最常见的用法是作为缓存层：将频繁查询但改动不频繁的数据（如用户信息、商品详情、配置、排行榜等）从数据库提前加载到Redis中，极大降低后端数据库压力，提高系统响应速度和吞吐量。
106	7.Spring Boot后端框架
107	Spring Boot是由Pivotal团队推出的开源Java后端框架，基于Spring生态并提供“开箱即用”的开发体验。它通过自动配置、约定优于配置和内嵌服务器等机制，显著减少繁琐的XML配置和环境搭建工作，使开发者能够快速构建独立运行、易部署的Web服务与微服务应用。本文后端采用Spring Boot实现业务逻辑与接口服务，结合数据库与缓存组件，完成数据处理、权限控制以及与前端的接口交互。
108	8.B/S架构
109	项目采用典型的 B/S 架构。前端由 Vue 3 + TypeScript + Vite 构建单页应用，后端由 Spring Boot 处理认证、业务规则、数据访问与实时推送，前后端通过 REST API 与 WebSocket 通信。这种方式部署成本低、迭代效率高，适合校园社区系统[4][8][9]。
110	后端采用 Controller / Service / Mapper 分层。Controller 处理接口与参数校验，Service 负责业务编排和权限，Mapper 基于 MyBatis-Plus 访问数据库；MySQL 负责持久化，Redis 负责缓存、会话和版本号管理。该结构降低了耦合度，也便于分别验证接口层和服务层逻辑。
111	结合本项目的业务特点，B/S 架构还承载了内容安全与智能处理能力：本地服务负责敏感词过滤、情感分析、热度计算和推荐组装，外部 AI 服务负责标签与摘要生成。通过把“本地稳定能力”和“外部增强能力”解耦，系统在保证业务连续性的同时，也保留了面向后续算法升级的扩展空间。

124	2.2 需求分析
125	本系统面向技术社区的日常交流与运营治理场景，核心需求是实现“安全发布、有效互动、智能分析、辅助决策”的一体化支撑。系统需覆盖用户从注册登录到内容消费的完整流程，并为管理员提供可执行的治理工具。同时，平台应具备基本扩展能力，在中小规模并发下保持稳定响应与可维护性。
126	1.功能需求分析
127	（1）用户与权限管理需求。系统需提供完整账号体系，包括注册、密码登录、邮箱验证码登录、找回密码与退出登录，并支持GitHub第三方登录以降低使用门槛。身份认证采用JWT无状态机制，结合Redis维护会话与设备信息，确保多端访问安全可控。权限模型需区分普通用户、版主、管理员等角色，不同角色在发帖管理、置顶加精、用户管理和后台配置上拥有差异化权限。系统还需支持二步验证，提升高权限账号安全等级；对未授权访问、令牌失效和越权操作返回统一错误信息，保证权限链路清晰可追踪。
128	（2）内容发布与互动需求。系统应支持帖子创建、编辑、删除、分页检索与详情查看，支持标签、摘要、图片、匿名发布及板块归类。互动功能需覆盖评论与回复、点赞、收藏、关注、私信和站内通知，形成基本社交闭环。平台需记录浏览行为并实时更新帖子互动指标，为后续热度计算与推荐提供数据基础；同时支持置顶、分类置顶、精华等内容曝光能力，满足社区运营需要。
129	（3）智能分析与推荐需求。系统需对帖子文本与行为数据进行分析，提供标签提取、摘要生成、情感打分、热度排行、关键词词云与趋势统计等能力。推荐模块应结合用户兴趣标签与内容热度进行混合推荐，在冷启动或数据不足时可回退到热门内容，保证推荐结果稳定可用，并为后续协同过滤扩展预留接口。
130	（4）后台治理与运维需求。系统需提供面向管理端的治理功能，包括用户封禁/解封、角色分配、版块管理、举报处理与通知和内容管理、缓存管理，确保社区秩序可控。内容治理层需具备敏感词过滤与违规文本拦截能力，降低不良内容传播风险。运维侧需支持缓存统计与按模式清理、关键日志记录、定时任务调度（如热度刷新与历史日志清理），并具备基础可观测性。为提升接口安全，系统还应支持请求签名、时间戳与随机串校验、防重放机制及跨域白名单配置。通过治理与运维能力协同，保障平台长期稳定运行。
131	2.非功能需求分析
132	（1）性能需求。系统需在中小规模并发访问场景下保持稳定响应，登录、帖子列表、帖子详情与推荐等核心接口应具备较低延迟，保证用户交互流畅。平台应具备稳定的并发处理能力，能够支持缓存访问、数据库读写、消息推送及定时任务并行执行，并避免后台计算对主业务流程造成明显阻塞。同时，系统应通过分页加载、数据访问优化与资源压缩等方式提升整体访问效率，为后续功能扩展预留性能余量。
133	（2）安全需求。系统需具备完整安全防护链路，包括JWT认证、基于角色的权限控制、会话校验、设备绑定、请求签名与防重放机制。对敏感操作应进行身份核验与权限拦截，对未授权访问、令牌过期和越权请求返回统一错误。平台还需进行敏感词过滤与内容治理，降低违规信息传播风险，保障账号与数据安全。
134	（3）可用性与易用性需求。系统界面与交互流程应符合论坛使用习惯，保证用户可快速完成注册、发帖、评论、检索和消息查看。管理端功能布局应清晰，便于管理员进行用户管理、内容审核和运营维护。系统需具备良好的兼容性与错误提示机制，在异常场景下提供可理解反馈，提升整体使用体验与操作效率。
135	（4）可维护性与可扩展性需求。系统应采用分层解耦设计，业务模块边界清晰，便于后续功能迭代和问题定位。代码需具备统一接口规范、日志规范与配置化能力，支持多环境部署与参数调整。推荐、趋势分析、AI 内容处理等模块应预留扩展接口，便于后续引入更复杂算法或第三方服务，保障系统长期演进能力。
136	3.性能保障机制分析
137	Campus Pulse 面向中小规模校园内容社区，性能设计更强调稳定可用、可观察和可扩展，而不是单纯追求峰值吞吐。结合当前实现，系统从请求监控、连接池、缓存一致性、定时任务和前端构建五个方面建立了性能保障机制。
138	（1）接口响应时间
139	在接口响应层，后端运行于 Spring Boot 3.5.8 内嵌 Tomcat 上，服务端口为 7800。系统通过 RequestTimingFilter 记录请求耗时，对慢请求输出告警日志；数据库访问链路则由 SlowSqlInterceptor 监控 SQL 执行时间，便于定位性能热点[4][9][15]。
140	根据 application.yml 配置，Tomcat 最大工作线程数为 200，最小空闲线程数为 20；前端统一配置请求超时与刷新令牌机制，保证弱网和高峰场景下的基本交互稳定性。该设计不追求过度拆分，而是通过清晰分层和关键链路监控控制系统复杂度。
141	（2）并发处理能力
142	并发处理方面，系统在后端使用 HikariCP 管理 MySQL 连接，最小空闲连接数为 8、最大连接数为 32；Redis 采用 Lettuce 连接池，最大活跃连接数为 20，最大等待时间为 3000 ms。两者分别承担持久化读写和高频缓存访问，能够满足帖子流、认证信息、通知统计与推荐结果等并发场景。
143	- Redis连接池配置：最大活跃连接数20个，最大空闲连接数10个，最小空闲连接数2个，连接获取最大等待时间3000ms。该配置采用Lettuce客户端驱动，支持帖子列表、帖子详情、热榜、推荐结果、用户令牌、验证码等多类缓存的并发读写。
144	- 数据库层面采用 HikariCP 连接池，最小空闲连接数配置为 8、最大连接数配置为 32，并通过 Actuator 暴露 HikariCP 监控指标，可实时观测连接池利用率。
145	实时通信层基于 STOMP over SockJS 构建 WebSocket 通道，后端提供 `/topic/posts`、`/topic/section/{id}` 与 `/user/queue/notifications` 等订阅目标，前端在通知中心、帖子列表和板块页执行局部刷新。相比轮询，该方式能减少重复请求并提升消息到达的即时性[5][15]。
146	缓存策略与数据一致性
147	系统采用 Redis 版本号键控制帖子流、帖子详情和热榜缓存失效。当帖子创建、评论新增、审核状态变化或互动计数变化时，业务层只需递增对应版本号即可使旧缓存自然过期，减少大规模主动删除的开销。针对空结果，系统还设置短 TTL 空值缓存以降低缓存穿透风险，并通过 TTL 抖动缓解集中失效带来的瞬时压力。
148	（4）定时任务与后台处理
149	系统包含以下定时任务，用于维护数据的时效性和准确性：
150	当前项目主要包含三类周期性任务：一是每 10 分钟执行一次帖子热度重算，针对最近 14 天内的帖子按照“浏览 × 0.5 + 点赞 × 3 + 评论 × 5 + 收藏 × 2”的互动权重与时间衰减函数重新计算热度；二是每小时执行一次标签热度衰减，避免历史高热标签长期占据热门区域；三是浏览日志清理与等级升级，用于维持统计数据时效性和表规模稳定。
151	额外加20分基础分，情感得分低于0.3的帖子施加0.7倍惩罚。仅当热度变化超过0.5时才执行数据库更新操作，减少不必要的写入。
152	服务端开启了 Gzip 压缩，对 JSON、HTML、CSS、JavaScript、SVG 等响应类型进行压缩传输，并设置 2 KB 以上响应启用压缩，以减少前后端通信时的带宽占用。该策略对帖子详情、趋势统计和后台列表等返回体较大的接口尤为有效。
153	（5）HTTP响应压缩
154	前端基于 Vite 构建，按框架库、编辑器库、图表库、WebSocket 依赖和工具依赖进行分块打包。通过静态资源拆分和浏览器缓存复用，用户再次访问时仅需重新加载变更 chunk，从而降低等待时间并提升趋势分析页和后台管理页的首屏稳定性。
155	（6）前端构建优化
156	前端基于Vite构建工具，采用代码分割（Code Splitting）策略，将第三方依赖拆分为独立的chunk：
157	- vue-vendor：Vue、Vue Router、Pinia、Element Plus 等核心框架库；
158	- editor-vendor：Markdown-it、DOMPurify 等编辑器相关依赖；
159	- chart-vendor：ECharts、Vue-ECharts 等图表库；
160	- ws-vendor：SockJS、STOMP.js 等 WebSocket 相关依赖；
161	- app-utils-vendor：Axios、date-fns、lucide-vue-next 等工具类依赖；
162	- misc-vendor：其他第三方依赖。
163	该策略利用浏览器缓存机制，使用户在后续访问时只需加载发生变化的代码块，显著提升页面二次加载速度。

1546	第4章 系统实现
1547	4.1 普通用户角色功能实现
1548	普通用户是 Campus Pulse 的主要使用主体。围绕“低门槛发布、可信身份、及时反馈”三个目标，前台提供注册登录、资料维护、发帖评论、点赞收藏、通知查看、私信会话和个性化推荐等完整链路，保证用户从首次进入到持续活跃都能在统一界面中完成主要操作。
1549	在认证与会话层，系统同时发放 accessToken 和 refreshToken，并将 sid、did 写入 JWT 声明；服务端再通过 Redis 的 `auth:access`、`auth:refresh` 和 `auth:device` 键空间校验会话与设备一致性。相比只校验令牌签名的方案，该设计更便于单设备控制、主动失效和异常登录提醒[10][14]。
1550	被窃取，只要设备不一致或会话已失效，网关过滤器也会拒绝请求。相比“只校验 JWT 签名”的传统方案，该机制明显提升了会话可控性。
1551	系统还提供基于 TOTP 的二步验证。用户在设置页初始化后可获得密钥、otpauth URI 与二维码；启用 2FA 的账号登录时必须额外提交动态验证码或先获取登录票据再完成确认。结合邮件验证码与 GitHub OAuth 登录，系统形成了多方式登录体系。
1552	对于 POST、PUT、PATCH 和 DELETE 等变更请求，后端 RequestSecurityFilter 会校验 `X-Request-Timestamp`、`X-Request-Nonce` 与 `X-Request-Signature`，并利用 Redis 记录随机串使用状态，防止重放提交。该方案实现成本较低，但能增强批量审核和资料修改等敏感接口的安全性[14]。
1553	/
1554	（1）内容发布与智能处理实现
1555	内容模块围绕发布、处理、展示和再编辑形成闭环，核心由 PostController 与 PostServiceImpl 协同完成。用户创建帖子时，系统会先校验板块与字段合法性，再执行敏感词处理、摘要构建、情感打分和标签整理，随后写入帖子表并触发缓存失效、事件广播和经验值更新。
1556	在内容安全方面，系统采用基于 DFA 思想的敏感词过滤服务，对标题、正文、评论及个人资料等文本字段进行扫描。一般违规词优先替换以保证发布链路连续性，需人工介入的内容再交由审核与举报流程处理。这种“轻拦截 + 后治理”的方式更适合校园社区场景。
1557	在智能处理方面，系统通过 `extract-tags` 与 `summary/regenerate` 接口接入外部 AI 服务，为帖子提供标签提取和摘要生成；当外部服务异常或未配置密钥时，后端回退到本地 Markdown 清洗与截断策略，保证发帖与编辑流程不中断[12]。
1558	此外，系统在本地实现了基于词典与分词的情感打分流程，并将分数持久化到帖子表中。热度计算时，负向情感得分过低的内容会受到降权处理，使热榜更偏向高互动、低风险和整体评价更稳定的内容[6][12]。
1559	帖子模型除基本字段外，还维护 `global_pin`、`category_pin`、`pin_order`、`pin_expire_at`、`last_reply_at` 和 `last_activity_at` 等运营字段，既支持全局置顶、板块置顶，也支持普通帖子按活跃度排序。
1560	/
1561	（2）社区互动与实时通信实现
1562	互动模块重点解决“反馈即时性”和“行为可沉淀”两类问题。系统实现了评论与回复、帖子点赞、帖子收藏、用户关注、站内通知和私信会话六类核心互动能力，并通过数据库计数、缓存版本更新和 WebSocket 推送把单次互动转换为可持续利用的行为数据。
1563	评论服务支持楼中楼回复、匿名评论和评论点赞；评论写入后会同步更新帖子 `comment_count`、`last_reply_at` 与 `last_activity_at` 并触发帖子流缓存失效。点赞与收藏服务采用幂等 toggle 逻辑，避免重复写入，并联动经验值、作者获赞数和通知消息。
1564	通知模块采用“数据库持久化 + WebSocket 推送 + 可选邮件同步”的三段式实现。通知落表后，系统通过 `convertAndSendToUser` 将未读消息推送至用户订阅队列，前端实时更新角标和通知列表；用户开启邮件同步时，再异步发送邮件提醒[5][15]。
1565	私信服务按双方用户 ID 有序拼接得到 conversationId，用于统一聚合同一对用户之间的双向消息。该建模方式简化了会话列表、分页查询、已读状态和未读统计的实现，也为后续扩展图片、语音等富媒体消息保留了接口空间。
1566	实时通信层基于 STOMP over SockJS 构建，后端向帖子总线、板块总线和用户通知队列推送事件；前端在帖子列表、趋势页和通知中心执行局部刷新。相比定时全量轮询，该方式能够降低重复请求量，并提升消息到达的即时性[5][15]。
1567	/
1568	（3）推荐分发与趋势浏览实现
1569	推荐分发模块采用“兴趣标签 + 协同过滤 + 热门兜底”的混合策略。登录用户先按标签关系召回兴趣帖子，再引入协同过滤结果，最后补充热门内容；匿名用户则直接返回热门推荐列表，以保证冷启动可用性[3][7][11][13]。
1570	帖子详情页相关推荐进一步采取“同专业用户内容 + 同板块内容 + 热门兜底”的三段式策略。该设计虽然不追求复杂模型，但能够在当前数据规模下有效提高相关性，并通过外置配置控制候选池大小、权重参数和分页数量，便于后续调优。
1571	热度计算任务每 10 分钟执行一次，核心公式以互动得分除以时间衰减因子，并叠加精华帖加权与低情感分惩罚。标签热度按小时衰减，避免历史高热标签长期垄断推荐与热榜区域，从而形成比单纯按时间排序更合理的分发逻辑[7][11][13]。
1572	从工程角度看，该模块的价值不在于引入最复杂的推荐算法，而在于把用户兴趣、内容质量和平台治理诉求放进同一条分发链路，使推荐结果既具有个性化特征，又不脱离社区整体运营目标。
1573	/
1574	4.2 版主角色功能实现
1575	版主角色主要负责板块级内容治理。系统根据版主申请审核结果获取其可管理的板块集合，并在帖子审核、举报处置、分类置顶、加精设置等接口中校验该集合，确保版主只对被授权板块拥有治理能力。
1576	在举报处理方面，系统实现了“待处理 - 排队中 - 处理中 - 已处理/忽略/打回修改”的异步流转机制。若举报指向帖子且需要作者修改，系统会同步更新帖子审核状态并向作者发送处理通知；若举报指向评论，则通过解析所属帖子板块完成权限判定。该设计提高了板块自治效率，也减少了管理员对普通板块事务的重复介入。
1577	通过板块级权限约束与独立工作台设计，版主能力不再依赖全局管理员逐条处理，而是以“分区自治 + 分级审核”的方式提升社区治理效率，也降低了后台集中运维压力。
1578	/
1579	4.3 管理员角色功能实现
1580	管理员承担站点级运营与运维职责，前端后台提供用户管理、板块管理、举报管理、版主申请管理、邀请码管理、缓存管理和仪表盘等页面；后端对应开放统计、缓存和内容治理接口，用于完成全站配置与数据维护。
1581	数据运维方面，管理员可以查看社区总览、热榜排行、板块分布、关键词聚合和趋势预测结果，也可以对 Redis 缓存执行概览、计数和批量清理。配合热度衰减、浏览日志清理和等级升级等后台任务，系统能够在持续运行中维持数据时效性与资源健康。
1582	总体而言，Campus Pulse 在角色设计上形成了“普通用户负责内容生产与互动、版主负责分区治理、管理员负责全局配置与运维”的三级协同结构。这种分工方式既符合社区类系统的管理习惯，也便于后续扩展更细粒度的权限模型。

1586	5.1 测试目标与测试方法
1587	系统测试的目标不是简单证明“页面可以打开”，而是验证 Campus Pulse 在真实业务链路上的正确性、安全性和稳定性，重点检查认证、发布、互动、治理和推荐等核心模块是否与设计保持一致[8][9]。
1588	结合当前项目实现，测试工作采用“控制器参数校验 + 服务逻辑测试 + 页面/接口联调验证”的组合方式。前者利用 JUnit 5、Mockito 和 MockMvc 验证输入约束、权限边界和返回结构；后者结合浏览器与接口工具联调关键业务链路。
1589	1. 功能测试：验证注册登录、发帖评论、推荐分发、私信通知、版主管理和后台治理等功能是否能够按预期完成业务流转。
1590	2. 接口测试：验证 REST API 的参数校验、状态码封装、权限拦截与异常提示是否一致。
1591	3. 服务测试：验证推荐服务、版主管理、通知批处理、摘要生成等关键业务逻辑在边界条件下的行为。
1592	4. 安全测试：验证 JWT 认证、刷新令牌、设备绑定、请求签名、防重放和二步验证相关流程是否有效。
1593	5. 兼容与性能观察：验证高频页面在 Chrome 和 Edge 中的基本兼容性，并通过日志与监控信息观察缓存、数据库和定时任务的运行状态。
1594	从已有测试代码看，项目测试主要覆盖四类场景：认证参数校验、内容发布约束、治理与通知处理以及文本摘要生成。对应的 `AuthControllerValidationTest`、`PostControllerValidationTest`、`NotificationControllerBatchTest`、`ModeratorApplicationControllerTest`、`SectionModeratorServiceImplTest` 和 `SummaryGenerationTest`，验证了刷新令牌、字段长度、批量操作、板块权限和摘要生成等关键逻辑。
1595	因此，测试章节不再使用与本项目无关的“教务流程”“选课模块”等描述，而是完全围绕当前代码仓库中的真实测试对象展开。重点检查输入是否合法、权限是否越权、状态是否一致、异常是否可预期，以及缓存和实时通知参与时系统能否稳定返回正确结果。
1596	安全边界测试：由于系统涉及学生的真实档案与成绩等敏感教务数据，测试需重点验证 JWT（JSON Web Token） 鉴权机制的有效性，确保未授权用户无法越权访问管理员运维接口。
1597	性能与稳定性评估：验证在校园突发热点引发的高并发场景下，系统通过 Redis 缓存热度排行（HeatRank）的响应时效性，确保系统在高负载下不崩溃、数据不丢失。
1598	智能化准确性校验：通过对不同类型的校园文本进行输入测试，验证系统集成 HanLP 进行自动标签提取的准确率，保障推荐引擎的数据源质量。
1599	2.测试方法
1600	本项目以黑盒测试为主、辅以白盒式控制器校验。前端页面与接口联调从用户视角输入合法或非法数据，观察输出是否满足业务要求；后端则通过 MockMvc、Mockito 和断言库验证边界值、异常路径和权限分支。
1601	等价类划分法：在认证与发帖模块中，将账号、邮箱、验证码、标题和正文长度等输入划分为有效与无效两类，并分别验证系统是否返回统一且可理解的错误提示。
1602	边界值分析法：针对帖子标题长度、正文最小长度、批量通知 ID 列表、版主申请理由长度等边界受限参数进行测试，防止因极端输入引发空指针、越界或错误落库。
1603	接口回归测试：当帖子模块、通知模块或版主管理模块发生改动时，重新执行既有控制器测试，确认新功能未破坏已有参数校验和权限约束。
1604	异常路径测试：模拟 refreshToken 为空、用户未登录、普通用户访问管理员接口、批量参数为空等异常输入，观察系统是否按统一结构返回错误并阻断后续业务执行。

1705	结  论
1706	系统实现与技术架构总结
1707	本文围绕 Campus Pulse 的真实开发过程，完成了一个面向校园内容交流与趋势分析场景的智能社区平台。系统以 Spring Boot 3.5.8、MyBatis-Plus、MySQL、Redis 与 Vue 3 + TypeScript 为核心技术栈，实现了账号认证、内容发布、评论回复、点赞收藏、私信通知、版主治理、趋势展示和后台管理等主要功能。
1708	与单纯按时间排序的论坛相比，Campus Pulse 更强调内容生产、行为采集、数据计算和治理反馈的联动。通过用户、帖子、互动、举报、通知和趋势统计等多域数据统一建模，系统既改善了用户体验，也为版主和管理员提供了辅助决策依据。
1709	2. 开发难点与关键技术解决方案
1710	系统开发过程中，第一类突出问题是高频访问下的数据一致性。为此，项目采用 Redis 版本化缓存、空值缓存和 TTL 抖动策略，在保证读取性能的同时减少缓存穿透和集中失效带来的波动。
1711	第二类是智能处理链路的稳定性问题。由于标签提取与摘要生成依赖外部 AI 服务，系统采用“外部能力优先、本地摘要兜底”的方式，确保第三方服务异常时核心发布流程仍可用。
1712	第三类问题是社区开放性与平台治理之间的平衡。项目通过匿名展示与后台可审计追踪并行的方式处理表达自由与治理责任，并通过 JWT、设备绑定、请求签名、角色权限和二步验证构建分层安全链路。
1713	最后，在社区活跃与平台监管之间，系统通过匿名显示与真实身份分离机制实现前台匿名表达，同时保留后台可审计追踪能力；并结合Spring Security、JWT、角色权限校验与请求防重放机制，构建了较完整的安全防护链路。
1714	3. 系统完善性与未来扩展方向
1715	尽管当前系统已经完成毕业设计阶段的预期目标，但在智能化深度和工程化能力上仍有继续完善的空间。
1716	第一，可以将当前基于角色和板块范围的权限判定进一步演进为更细粒度的 RBAC/ABAC 混合模型，补充资源级授权、行为审计与风险告警能力。
