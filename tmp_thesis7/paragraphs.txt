48	[2]	Abstract
49	[]	With the rapid development of open-source ecosystems, online learning platforms and technology sharing platforms, technical communities have become an important infrastructure for developers to acquire knowledge and exchange experiences. Many small and medium-sized communities lack unified data analysis and decision support mechanisms, making it difficult for them to continuously improve user experience and community health. In this context, researching intelligent content analysis and decision support systems for technical communities can provide data-based assistance for community operations and moderators, reduce subjective judgment biases, and improve content governance quality and platform operation efficiency.
50	[]	The system is built based on Spring Boot, uses Spring Security + JWT to achieve stateless authentication and permission control, and introduces Redis to support session verification, request prevention of replay, recommendation results and post stream caching, improving the stability and response efficiency of the system under concurrent access. The system realizes functions such as user registration and login, post publication search, comment interaction, likes and favorites, private message notifications, and personal profile viewing and modification, and implements core functions such as managing board categories, post content, reporting content, and development history in the administrator backend, forming a sustainable running community business loop.

85	[3]	1.1 研究背景与意义
86	[]	随着开源生态、在线学习与技术分享平台的快速发展，技术社区已成为开发者获取知识、交流经验和协作创新的重要基础设施。与传统论坛相比，当前社区内容呈现出更新频率高、主题跨度大、结构化程度低的特点，海量帖子、评论与互动数据在带来活跃度的同时，也引发了信息过载、优质内容被淹没、低质量信息扩散和管理成本上升等问题。许多中小型社区仍以人工经验进行内容筛选与运营判断，缺乏统一的数据分析与决策支持机制，导致热点识别滞后、推荐相关性不足、治理效率不高，难以持续提升用户体验与社区健康度。在此背景下，研究面向技术社区的智能内容分析与辅助决策系统，具有明确的现实需求与工程价值。
87	[]	本课题的意义主要体现在三个层面。第一，在业务层面，通过构建“内容发布—行为采集—趋势分析—推荐分发—治理反馈”的闭环机制，可实现对社区动态的持续感知，提升管理者对热点演化、内容质量与用户兴趣变化的把握能力。第二，在技术层面，将身份认证与权限控制、缓存加速、敏感内容过滤、情感分析、热度衰减计算与混合推荐策略进行一体化落地，为中小规模技术社区提供可复用的轻量化实现路径，兼顾安全性、实时性与可维护性。第三，在应用层面，系统不仅服务普通用户的高效获取与互动，还能为运营与版主提供数据化辅助决策依据，减少主观判断偏差，提升内容治理质量与平台运行效率。因而，本研究对推动技术社区从“功能型论坛”向“数据驱动的智能协作平台”转型具有积极意义。
88	[3]	1.2 研究内容
89	[]	本课题围绕技术社区“内容管理、趋势洞察与辅助决策”三条主线展开研究与实现。首先，在系统架构层面，构建基于Spring Boot 3的前后端分离论坛平台，完成用户注册登录、帖子发布检索、评论互动、点赞收藏、私信通知、版块与后台管理等核心功能，形成可持续运行的社区业务闭环。
90	[]	其次，在安全与性能层面，采用Spring Security + JWT实现无状态认证与权限控制，引入Redis支持会话校验、请求防重放、推荐结果与帖子流缓存，提升系统在并发访问下的稳定性与响应效率。
91	[]	再次，在智能分析层面，针对社区文本与行为数据，设计并实现敏感词过滤、情感分析、AI标签/摘要提取、关键词词云统计与热度排行；构建“互动权重+时间衰减+情感惩罚”的热度计算机制，用于热点识别与趋势展示。最后，在个性化服务层面，结合用户兴趣标签与内容热度实现轻量化混合推荐，并为协同过滤扩展预留接口。通过上述研究，系统实现了从数据采集、内容分析到推荐分发与治理反馈的完整链路，为技术社区运营提供可落地的辅助决策支撑。

123	[3]	2.2需求分析
124	[]	本系统面向技术社区的日常交流与运营治理场景，核心需求是实现“安全发布、有效互动、智能分析、辅助决策”的一体化支撑。系统需覆盖用户从注册登录到内容消费的完整流程，并为管理员提供可执行的治理工具。同时，平台应具备基本扩展能力，在中小规模并发下保持稳定响应与可维护性。
125	[]	1.功能需求分析
126	[]	（1）用户与权限管理需求。系统需提供完整账号体系，包括注册、密码登录、邮箱验证码登录、找回密码与退出登录，并支持GitHub第三方登录以降低使用门槛。身份认证采用JWT无状态机制，结合Redis维护会话与设备信息，确保多端访问安全可控。权限模型需区分普通用户、版主、管理员等角色，不同角色在发帖管理、置顶加精、用户管理和后台配置上拥有差异化权限。系统还需支持二步验证，提升高权限账号安全等级；对未授权访问、令牌失效和越权操作返回统一错误信息，保证权限链路清晰可追踪。
127	[]	（2）内容发布与互动需求。系统应支持帖子创建、编辑、删除、分页检索与详情查看，支持标签、摘要、图片、匿名发布及板块归类。互动功能需覆盖评论与回复、点赞、收藏、关注、私信和站内通知，形成基本社交闭环。平台需记录浏览行为并实时更新帖子互动指标，为后续热度计算与推荐提供数据基础；同时支持置顶、分类置顶、精华等内容曝光能力，满足社区运营需要。
128	[]	（3）智能分析与推荐需求。系统需对帖子文本与行为数据进行分析，提供标签提取、摘要生成、情感打分、热度排行、关键词词云与趋势统计等能力。推荐模块应结合用户兴趣标签与内容热度进行混合推荐，在冷启动或数据不足时可回退到热门内容，保证推荐结果稳定可用，并为后续协同过滤扩展预留接口。
129	[]	（4）后台治理与运维需求。系统需提供面向管理端的治理功能，包括用户封禁/解封、角色分配、版块管理、举报处理与通知和内容管理、缓存管理，确保社区秩序可控。内容治理层需具备敏感词过滤与违规文本拦截能力，降低不良内容传播风险。运维侧需支持缓存统计与按模式清理、关键日志记录、定时任务调度（如热度刷新与历史日志清理），并具备基础可观测性。为提升接口安全，系统还应支持请求签名、时间戳与随机串校验、防重放机制及跨域白名单配置。通过治理与运维能力协同，保障平台长期稳定运行。
130	[]	2.非功能需求分析
131	[]	（1）性能需求。系统需在中小规模并发访问场景下保持稳定响应，登录、帖子列表、帖子详情与推荐等核心接口应具备较低延迟，保证用户交互流畅。平台应具备稳定的并发处理能力，能够支持缓存访问、数据库读写、消息推送及定时任务并行执行，并避免后台计算对主业务流程造成明显阻塞。同时，系统应通过分页加载、数据访问优化与资源压缩等方式提升整体访问效率，为后续功能扩展预留性能余量。
132	[]	（2）安全需求。系统需具备完整安全防护链路，包括JWT认证、基于角色的权限控制、会话校验、设备绑定、请求签名与防重放机制。对敏感操作应进行身份核验与权限拦截，对未授权访问、令牌过期和越权请求返回统一错误。平台还需进行敏感词过滤与内容治理，降低违规信息传播风险，保障账号与数据安全。
133	[]	（3）可用性与易用性需求。系统界面与交互流程应符合论坛使用习惯，保证用户可快速完成注册、发帖、评论、检索和消息查看。管理端功能布局应清晰，便于管理员进行用户管理、内容审核和运营维护。系统需具备良好的兼容性与错误提示机制，在异常场景下提供可理解反馈，提升整体使用体验与操作效率。
134	[]	（4）可维护性与可扩展性需求。系统应采用分层解耦设计，业务模块边界清晰，便于后续功能迭代和问题定位。代码需具备统一接口规范、日志规范与配置化能力，支持多环境部署与参数调整。推荐、趋势分析、AI 内容处理等模块应预留扩展接口，便于后续引入更复杂算法或第三方服务，保障系统长期演进能力。
135	[]	3.性能保障机制分析
136	[]	Campus Pulse 技术社区平台面向中小规模用户群体。为满足上述性能需求，系统结合当前技术架构，从接口监控、并发处理、缓存一致性、定时任务调度、响应压缩和前端构建优化等方面构建了性能保障机制。
137	[]	（1）接口响应时间
138	[]	系统采用 Spring Boot 3.5.8 框架，内置 Tomcat 作为应用服务器，服务端口为 7800。为保障用户交互的流畅性，系统对 HTTP 请求链路与数据库访问链路进行分层监控。
139	[]	系统通过 RequestTimingFilter 对每个 HTTP 请求进行耗时监控，当接口响应时间超过 1000ms 时自动记录为慢请求并输出 WARN 级别日志。同时，通过 SlowSqlInterceptor 对 MyBatis 执行的 SQL 语句进行监控，当单条 SQL 执行时间超过 300ms 时记录为慢 SQL。前端 Axios 客户端统一设置请求超时时间为 15 秒，刷新令牌请求超时时间为 12 秒，超时后自动中断并提示用户。
140	[]	（2）并发处理能力
141	[]	系统基于 Spring Boot 内嵌 Tomcat 线程池、Redis 连接池与数据库连接池协同处理并发请求。其中，Tomcat 最大工作线程数配置为 200、最小空闲线程数为 20，并通过缓存与连接池配合提升请求承载能力：
142	[]	  - Redis连接池配置：最大活跃连接数20个，最大空闲连接数10个，最小空闲连接数2个，连接获取最大等待时间3000ms。该配置采用Lettuce客户端驱动，支持帖子列表、帖子详情、热榜、推荐结果、用户令牌、验证码等多类缓存的并发读写。
143	[]	- 数据库层面采用 HikariCP 连接池，最小空闲连接数配置为 8、最大连接数配置为 32，并通过 Actuator 暴露 HikariCP 监控指标，可实时观测连接池利用率。
144	[]	  - WebSocket长连接基于STOMP协议和SockJS降级方案实现，支持全局帖子事件广播（/topic/posts）、板块级事件订阅（/topic/section/{id}）和用户个人通知推送（/user/{id}/queue/notifications）。WebSocket客户端采用指数退避重连策略，基础延迟1500ms，最大延迟30000ms，最多递增6次指数。
145	[]	缓存策略与数据一致性
146	[]	系统采用 Redis 作为分布式缓存层，对帖子列表、帖子详情与热榜分别维护版本号键，并在缓存 Key 中拼接版本号实现失效控制。当帖子、评论、浏览或审核结果发生变化时，业务层递增对应版本号，使旧缓存自动失效，在减少主动删除成本的同时兼顾读性能与数据一致性。
147	[]	（4）定时任务与后台处理
148	[]	系统包含以下定时任务，用于维护数据的时效性和准确性：
149	[]	  - 帖子热度衰减任务：每10分钟执行一次，采用改良版Hacker News算法（heatScore = interactionScore / (1 +ageInHours/24)^1.5），对最近14天内创建的帖子进行热度重新计算。互动得分权重为：浏览×0.5、点赞×3、评论×5、收藏×2。精华帖
150	[]	额外加20分基础分，情感得分低于0.3的帖子施加0.7倍惩罚。仅当热度变化超过0.5时才执行数据库更新操作，减少不必要的写入。
151	[]	  - 浏览日志清理：系统保留最近90天的浏览日志数据，超期数据定期清理以控制sys_view_log表的数据量增长。
152	[]	（5）HTTP响应压缩
153	[]	服务端开启了Gzip压缩，对大于2KB的JSON、XML、HTML、CSS、JavaScript及SVG等响应内容进行压缩传输，有效减少网络带宽占用，提升前端页面加载速度。

303	[2]	第4章 系统实现
304	[]	4.1 普通用户角色功能实现
305	[]	普通用户是 Campus Pulse 的主要使用主体，其典型使用链路包括注册登录、资料维护、发帖评论、点赞收藏、私信通知与推荐浏览。系统前台围绕“低门槛参与、身份可信、反馈及时”三个目标展开设计，首先从账号认证与会话安全切入。
306	[]	在会话管理方面，系统并非仅发放 accessToken，而是同时发放 refreshToken，并将会话标识（sid）与设备标识（did）写入令牌声明。服务端在 Redis 中维护 auth:access、auth:refresh、auth:device 等键空间，实现会话级校验与主动失效能力。这样即使令牌
307	[]	被窃取，只要设备不一致或会话已失效，网关过滤器也会拒绝请求。相比“只校验 JWT 签名”的传统方案，该机制明显提升了会话可控性。
308	[]	系统还实现了基于 TOTP 的二步验证流程。用户可在设置页发起 2FA 初始化，系统生成密钥、otpauth URI 与二维码，用户在认证器App 完成绑定后，登录阶段将触发二次动态码校验。这一设计在不显著增加交互复杂度的前提下，提高了高价值账号的防护等级。
309	[]	此外，请求安全层引入了时间戳、随机串（nonce）、签名校验机制。对于 POST/PUT/DELETE 等变更请求，过滤器可校验 X-Request-Timestamp、X-Request-Nonce、X-Request-Signature，并用 Redis 防重放。该机制可有效抑制接口重放攻击和部分中间人篡改风险。
311	[3]	（1）内容发布与智能处理实现
312	[]	内容模块是系统主干，围绕“发布—处理—展示—再编辑”形成闭环。后端通过 PostController、PostServiceImpl 实现帖子创建、检索、更新、删除、置顶、加精等全链路操作。用户发布帖子时，系统首先完成板块合法性校验，再执行文本合规处理与智能分析，最终写入结构化数据模型。
313	[]	在内容安全方面，系统使用 DFA 敏感词算法（ContentSecurityServiceImpl）对标题、正文、评论、个人资料字段进行扫描。命中词条后采用替换策略（*）处理，既保证发布流程连续性，又能降低违规内容传播风险。该策略在校园社区场景下具有较高实用价值：一方面避免“全拦截”导致用户体验断层，另一方面实现风险内容的可控降噪。
314	[]	在智能处理方面，系统接入 AI 服务用于标签提取与摘要重新生成。发帖与摘要重新生成功能分别通过 extract-tags 与 summary/regenerate 接口暴露。若外部 AI 服务异常，系统自动回退到本地摘要策略（基于正文截断与 Markdown 清洗），保证业务稳定性。
315	[]	同时系统对帖子执行情感分析（正向/中性/负向），将情感得分写入数据表并参与后续热度计算与趋势标注。
316	[]	  帖子模型除基本字段外，还扩展了 globalPin、categoryPin、pinOrder、pinExpireAt、lastReplyAt、lastActivityAt 等字段。结合检索接口中的排序规则，系统能够实现“全局置顶优先、板块置顶次之、普通帖子按活跃时间倒序”的信息流组织方式，保证重要信息和活跃讨论都能获得合适曝光。
318	[3]	（2）社区互动与实时通信实现
319	[]	互动模块重点解决“反馈即时性”和“行为可沉淀”两类问题。系统实现了评论、点赞、收藏、关注、私信、通知六类核心互动能力，并通过 WebSocket 增强实时感知。
320	[]	评论能力由 CommentServiceImpl 实现，支持楼中楼回复、匿名评论与评论点赞。评论写入后系统会同步更新帖子 commentCount、lastReplyAt、lastActivityAt，并触发帖子流缓存版本递增，保证列表数据快速一致。点赞与收藏由 PostLikeServiceImpl、PostCollectServiceImpl 提供 toggle 逻辑，避免重复写入，并同步更新帖子计数、作者获赞统计、经验值与通知消息。
321	[]	   通知模块（NotificationServiceImpl）同时支持站内与站外触达。站内通知通过数据库持久化并提供未读统计；站外通知在用户开启邮箱同步时异步发送邮件。触发类型包括评论、点赞、收藏、关注、@提及，形成完整的社交反馈闭环。前端可在通知页进行“单条已读、全部已读、删除”等管理操作。
322	[]	  系统还实现私信会话（DirectMessageServiceImpl）：会话按双方用户ID有序拼接生成 conversationId，支持会话列表、消息分页、会话已读和全局未读数统计。该设计既简化了建模，也便于后续扩展“消息撤回/多媒体消息”等功能。
323	[]	   实时通信层采用 STOMP over SockJS。后端向 /topic/posts、/topic/section/{id} 推送帖子事件（新帖、回复、浏览、点赞、置顶更新等），前端在 TopicList 中消费事件并执行局部刷新或排序前移；当 WebSocket 不可用时自动降级轮询，保证实时能力具备“可增强、可退化”的工程韧性。
325	[3]	（3）推荐分发与趋势浏览实现
326	[]	面对校园社区信息过载问题，系统在推荐层采用“个性化 + 协同过滤 + 热门兜底”的混合策略。RecommendController 对外提供推荐列表和详情页相关推荐接口，核心实现位于 PostRecommendServiceImpl。对于登录用户，系统综合用户兴趣标签权重、协同过滤结果与热门内容生成候选集，再按综合得分排序去重；对于匿名用户，则直接返回热门推荐，保证冷启动阶段也有可读内容。
327	[]	详情页相关推荐进一步引入“同专业用户内容 + 同板块内容 + 热门兜底”三段式策略，提升语义邻近性和场景相关性。推荐参数（如候选池大小、各类来源权重）通过配置类外置，便于论文中的参数调优实验与对比分析。
328	[]	热度演化方面，系统通过 PostHeatDecayTask 周期性执行热度重算。算法核心遵循“互动得分 / 时间衰减函数”，并加入“负向情感惩罚”和“精华帖加权”机制：既鼓励高质量高互动内容，也抑制消极内容持续霸榜。标签维度同步引入 TagHeatDecayTask 做小时级衰减，避免历史标签长期垄断热门区域。
329	[]	  该模块使社区流量分发从“静态时间排序”升级为“动态质量排序”，兼顾实时性、公平性与内容质量导向。
331	[3]	4.2 版主角色功能实现
332	[]	版主角色聚焦板块级内容治理。前端通过 MyModerationPage 为版主展示其负责板块，后台内容管理页基于 moderatedSectionIds 自动限定可操作范围；后端则通过 /post/moderation-list、approve、reject、category-pin、feature 等接口完成板块帖子审核、置顶与加精处理。
333	[]	在举报处置方面，版主可进入 ReportsManagePage 查看本板块举报记录，对举报执行“已处理、忽略、打回修改”等操作。系统在打回帖子后会同步修改帖子审核状态，并通过通知服务将处理结果发送给作者，形成“发现问题 - 处置问题 - 反馈作者”的闭环流程。
334	[]	通过板块级权限约束与独立工作台设计，版主能力不再依赖全局管理员逐条处理，而是以“分区自治 + 分级审核”的方式提升社区治理效率，也降低了后台集中运维压力。
336	[3]	4.3 管理员角色功能实现
337	[]	管理员角色承担站点级运营与系统治理职责。前端后台提供 DashboardPage、UsersManagePage、SectionsManagePage、ModeratorApplicationsManagePage、InviteCodesPage、ChangelogManagePage 等页面，支持数据看板、用户管理、板块配置、版主申请审核、邀请码管理与发展历程维护等核心操作。

339	[]	权限控制目前采用基于角色代码与板块权限的组合判定逻辑（ROLE_ADMIN、ROLE_SUPER_ADMIN 以及 moderatedSectionIds），并在关键管理接口中进行显式校验，确保高危操作具备访问边界。整体上，管理员角色负责全局配置、权限治理、运营分析和系统维护，是平台稳定运行的最终保障层。

351	[]	针对 Campus Pulse 系统的测试，其核心目的并非仅停留于发现程序中的语法错误，而是从业务逻辑、数据安全及系统稳定性三个维度进行综合验证：
352	[]	功能完整性验证：确保从会员端的“智能化内容发布”、“个性化推荐展示”到管理员端的“教务流程审批”等核心业务链路在逻辑上完全闭合，确保接口返回数据符合 RESTful 规范。
353	[]	安全边界测试：由于系统涉及学生的真实档案与成绩等敏感教务数据，测试需重点验证 JWT（JSON Web Token） 鉴权机制的有效性，确保未授权用户无法越权访问管理员运维接口。
354	[]	性能与稳定性评估：验证在校园突发热点引发的高并发场景下，系统通过 Redis 缓存热度排行（HeatRank）的响应时效性，确保系统在高负载下不崩溃、数据不丢失。
355	[]	智能化准确性校验：通过对不同类型的校园文本进行输入测试，验证系统集成 HanLP 进行自动标签提取的准确率，保障推荐引擎的数据源质量。
356	[]	2.测试方法
357	[]	本系统开发过程中采用了**黑盒测试（Black-box Testing）**作为主要的测试手段。黑盒测试不关注代码内部的逻辑实现，而是将系统看作一个“黑匣子”，通过模拟用户行为进行输入，观察输出结果是否符合预期需求。针对 CampusPulse 的系统特性，具体执行策略如下：
358	[]	等价类划分法： 在“认证与安全”模块测试中，将用户输入的邮箱、学号及密码分为“有效等价类”与“无效等价类”。例如，在注册接口测试时，分别输入格式正确的邮箱后缀与非法字符，验证系统拦截逻辑是否能给出统一的错误反馈，确保后台验证器的健壮性。
359	[]	边界值分析法： 针对“帖子管理”与“评论系统”，对内容长度限制、分页查询的页码边界（如第 0 页或极大页码）进行极端测试。在“教务选课”模块中，针对选课人数上限等临界点进行反复推演，防止由于边界值处理不当导致的逻辑溢出或数据库死锁。
360	[]	接口回归测试： 利用 Postman 自动化脚本对已实现的 RESTful API 进行批量调用。在对某个模块（如请假流程）进行优化或代码重构后，重新运行全量接口测试用例，确保新功能的加入未对现有的用户画像、推荐列表等存量功能造成干扰。
361	[]	异常路径测试（容错性测试）： 模拟网络波动、Token 伪造、Redis 服务宕机等异常场景。测试系统在失去缓存支撑时，能否通过 MySQL 数据库实现实时数据的“降级”展示，验证系统在极端情况下的自我恢复能力与运维治理接口的响应速度。

402	[2]	结  论
403	[]	系统实现与技术架构总结
404	[]	本研究围绕高校社区场景设计并实现了 CampusPulse 系统，形成了一套集“内容生产、互动反馈、智能推荐、趋势分析、平台治理”于一体的后端与前端协同平台。系统后端基于 Java 17 与 Spring Boot 3.x 构建，采用 MyBatis-Plus 完成数据持久化，结合 Redis实现高频数据缓存与会话状态管理；前端基于 Vue 3 + TypeScript 实现社区交互与管理可视化页面。在能力层面，系统已实现基于 JWT 的认证鉴权与刷新机制、设备绑定与二步验证、敏感词过滤与情感分析、AI 标签/摘要生成、混合推荐、热度衰减排行、站内通知与私信会话、后台内容治理与缓存运维等核心功能。通过对“用户-内容-行为-统计-治理”多域数据的统一建模，系统有效缓解了传统校园系统中业务割裂、响应滞后和可观测性不足的问题。综合测试结果表明，系统功能链路完整、接口行为稳定、权限边界清晰，能够满足校园社区平台在日常运行中的业务需求，并具备较好的工程可扩展性。
405	[]	通过对“用户-内容-互动-统计-教务”五大业务域的逻辑抽象，本系统不仅打破了传统校园应用中社交数据与行政数据孤立的局面，还利用 Redis 高速缓存技术保障了热度排行与 Token 鉴权的高并发响应能力。经测试，系统各功能接口均符合预期设计要求，能够稳定支撑学生画像构建、个性化内容推荐及趋势可视化分析等业务逻辑。
406	[]	2. 开发难点与关键技术解决方案
407	[]	系统实现过程中，主要难点集中在高并发场景下的数据一致性、智能处理链路的稳定性以及安全与匿名的平衡控制三个方面。
408	[]	首先，在热度排行与帖子流展示中，若完全依赖数据库实时计算，容易出现查询开销大与响应抖动的问题。为此，系统采用“Redis 缓存 + 定时热度重算 + 版本化失效”的组合策略，在保证响应速度的同时维持数据最终一致性。
409	[]	其次，在智能处理环节，AI 标签与摘要能力存在外部服务波动风险。系统通过“AI 结果优先、本地摘要兜底”的容错设计，确保在外部能力异常时仍可维持业务连续性，避免核心发布链路被阻断。
410	[]	  最后，在社区活跃与平台监管之间，系统通过匿名显示与真实身份分离机制实现前台匿名表达，同时保留后台可审计追踪能力；并结合Spring Security、JWT、角色权限校验与请求防重放机制，构建了较完整的安全防护链路。
411	[]	3. 系统完善性与未来扩展方向
412	[]	尽管 CampusPulse 已完成毕业设计预期目标，但在精细化治理与智能化深度方面仍有进一步提升空间。
413	[]	第一，权限体系可由当前角色判定进一步演进为更细粒度的 RBAC/ABAC 模型，补充资源级授权与操作级审计，提升敏感操作的可追溯性与合规能力。
414	[]	第二，推荐系统目前以兴趣标签、协同过滤与热门兜底为主，后续可引入浏览停留时长、点击反馈、负反馈等行为信号，构建“曝光-点击-反馈”闭环模型，提高推荐准确率与新内容发现能力。
415	[]	第三，实时能力可继续增强，未来可在现有 WebSocket 推送基础上扩展更完整的事件总线与消息可靠投递机制，并完善移动端/小程序接入，提升多端一致体验与校园通知触达效率。
416	[]	总体而言，本课题完成了校园社区智能化平台的关键技术落地，验证了“内容社区 + 数据分析 + 安全治理”一体化架构在高校场景中的可行性，为后续工程化部署和持续迭代奠定了基础。
